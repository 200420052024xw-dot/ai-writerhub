# 功能模块与系统架构说明

本文说明项目中各功能模块的实现方式，以及系统级组件如何为这些模块提供服务。项目整体是一个前后端分离的 AI 文档工作台：前端负责编辑、交互、配置和流式结果展示；后端负责 API 编排、模型调用、文档存储、格式导出、知识库索引和检索增强问答。

## 一、系统整体结构

项目主目录下主要分为以下部分：

- `frontend/`：React + TypeScript + Vite 前端应用，负责页面、编辑器、设置、文件选择、问答和翻译等交互。
- `backend/`：FastAPI 后端应用，负责 REST/SSE API、AI 模型调用、SQLite 数据持久化、文档解析、RAG 索引和 Word 导出。
- `backend/app/storage/`：后端运行时数据目录，保存文档数据库、原始上传文件、临时转换文件和 Chroma 向量库。
- `fronted_picture/`：前端展示或文档使用的图片资源。
- `editorai_project/`：项目开发计划、阶段报告、上下文记录和实现说明资料。

系统的基本调用链是：

1. 用户在前端页面操作。
2. 前端页面通过 `frontend/src/services/api.ts` 调用后端 `/api/...` 接口。
3. 后端路由层接收请求并校验 Pydantic schema。
4. 后端服务层完成业务处理，例如调用模型、读写 SQLite、生成 docx、写入 Chroma。
5. 后端返回普通 JSON、文件流，或使用 Server-Sent Events 返回流式结果。
6. 前端根据返回结果更新页面状态、localStorage 缓存和编辑器内容。

## 二、前端系统级实现

### 1. 应用入口与健康检查

前端入口是 `frontend/src/App.tsx`。应用启动后调用 `checkHealth()` 请求后端 `/api/health`，将后端状态设置为 `checking`、`online` 或 `offline`，再传给 `AppShell` 展示。

这个健康状态主要服务于顶部状态栏，让用户知道后端服务是否可用。它不直接控制页面功能，但能提示 API 不可达的问题。

### 2. 全局布局与页面调度

`frontend/src/layouts/AppShell.tsx` 是前端的系统壳层，负责：

- 左侧导航栏：主页、编辑、翻译、知识库、格式整理、设置。
- 顶部栏：当前文档标题、保存状态、模型配置状态、文件选择入口。
- 页面状态管理：记录当前打开的编辑文档、翻译文档和格式整理文档。
- 文件选择弹层：调用文档列表接口，允许在不同功能页面中选择同一份文档。
- localStorage 缓存：把最近使用的文档保存到 `writerhub.editorDocument`、`writerhub.translateDocument`、`writerhub.formatDocument`。

因此，功能页面不需要自己管理全局导航和跨模块文档选择。它们只接收 `AppShell` 传入的当前文档，然后专注于自己的功能。

### 3. API 封装层

`frontend/src/services/api.ts` 是前端统一 API 层。它定义了：

- 后端地址 `API_BASE_URL`，默认是 `http://127.0.0.1:8000`，也支持通过 `VITE_API_BASE_URL` 覆盖。
- 文档、翻译、格式整理、知识库问答、历史会话、Markdown 检测等接口函数。
- 前后端共享的数据结构类型，例如 `StoredDocumentDetail`、`TranslationResponse`、`RagRuntimeConfig`、`KnowledgeConversation`。
- 流式接口解析逻辑，例如 `translateTextStream()` 和 `queryRagStream()`。

这个 API 层为所有页面提供统一服务，避免页面直接拼 URL 或重复解析 SSE 数据。

### 4. 设置持久化

模型设置由 `frontend/src/services/modelSettings.ts` 管理，保存到 localStorage 的 `writerhub.modelSettings`。它提供多种模型服务商预设，包括 OpenAI、DeepSeek、通义千问、智谱、火山方舟、小米 MiMo、MiniMax 和自定义配置。

RAG 设置由 `frontend/src/services/ragSettings.ts` 管理，保存到 localStorage 的 `writerhub.ragSettings`。配置项包括：

- embedding 来源：本地模型或 API。
- 本地 embedding 模型路径。
- embedding API key、base URL 和模型名。
- 召回策略：向量召回或混合召回。
- 是否启用 rerank。
- rerank 模型路径。

这些设置会被功能页面转换成后端需要的运行时配置，随请求一起发送给后端。

## 三、后端系统级实现

### 1. FastAPI 应用和路由挂载

后端入口是 `backend/app/main.py`。`create_app()` 创建 FastAPI 应用，配置 CORS，允许 Vite 前端的 `127.0.0.1:5173` 和 `localhost:5173` 访问，然后统一挂载以下路由：

- `/api/health`：健康检查。
- `/api/config/ai`：后端 AI 默认配置查询。
- `/api/markdown/detect`：Markdown 特征检测。
- `/api/translate`、`/api/translate/stream`：翻译。
- `/api/assistant/chat`：编辑助手对话。
- `/api/format/...`：格式解析、整理、docx 导出。
- `/api/documents/...`：文档管理、上传、索引。
- `/api/rag/query/stream`：知识库问答。
- `/api/knowledge/conversations` 和 `/api/documents/{id}/assistant-history`：历史会话。

路由层只做请求接收、schema 校验和服务调用，核心业务逻辑放在 `backend/app/services/`。

### 2. 配置系统

`backend/app/core/config.py` 使用 `pydantic-settings` 从 `.env` 和环境变量读取配置，并通过 `get_settings()` 做缓存。配置内容包括：

- 默认 AI 服务商、API key、base URL、模型名和超时时间。
- RAG embedding 来源、本地模型路径、API 配置。
- RAG 召回策略和 rerank 配置。

后端配置为没有前端传入运行时配置的场景提供默认值，尤其服务于 RAG 索引和检索。

### 3. LLM 调用统一客户端

`backend/app/services/llm_client.py` 是系统级 AI 接入层。它提供：

- `RuntimeModelConfig`：运行时模型配置，包含 `api_key`、`base_url`、`model`。
- `ensure_runtime_model_config()`：校验并标准化模型配置。
- `chat_completions_url()`：把 base URL 规范化为 `/chat/completions` 端点。
- `call_chat_model()`：普通非流式文本模型调用。
- `stream_chat_model()`：流式模型调用，直接转发供应商 SSE 数据。
- `call_chat_messages()`：多轮 messages 调用。
- `call_vision_model()`：图像理解模型调用，用于上传文档的 OCR/版面识别。
- `test_chat_model()`：模型连通性测试。

翻译、格式整理、编辑助手、RAG 回答和文档识别都通过这一层调用模型，因此模型鉴权、URL 规范化、异常转换和特殊供应商适配集中在一个地方。

### 4. 数据校验与传输 schema

`backend/app/schemas/` 定义 Pydantic 模型，负责请求和响应结构。每个业务模块都有独立 schema，例如：

- `translation.py`：翻译方向、显示模式、术语表、翻译块、段落对齐结果、文档翻译状态。
- `documents.py`：文档摘要、文档详情、段落结构、创建和更新请求。
- `rag.py`：RAG 配置、检索结果、问答请求。
- `format.py`：格式配置、文档块、导出请求。
- `assistant.py`：助手消息和聊天请求。
- `chat_history.py`：知识库对话历史和文档助手历史。

schema 的作用是让路由层和前端 API 类型保持一致，并把复杂业务结果变成稳定的数据结构。

### 5. 存储系统

后端主要使用三类存储：

- SQLite：`backend/app/storage/documents/documents.sqlite3`，保存文档、段落、术语表、RAG chunk 元数据、全文检索表、问答会话、助手历史和翻译状态。
- 文件目录：`backend/app/storage/documents/originals` 保存上传原始文件，`backend/app/storage/documents/work` 保存转换后的临时图片或 PDF。
- Chroma：`backend/app/storage/chroma` 保存向量索引，用于语义检索。

`document_service.ensure_storage()` 会创建文档相关表，`rag_service.ensure_rag_tables()` 会创建 RAG chunk 表和 FTS5 全文检索表，`chat_history_service.ensure_chat_history_tables()` 和 `translation_state_service.ensure_translation_state_table()` 分别创建历史和翻译状态表。

这些存储能力支撑了文档长期保存、知识库索引、历史会话恢复和跨页面状态复用。

### 6. Prompt 系统

`backend/app/prompts/` 存放不同业务的提示词构造逻辑：

- `translation.py`：翻译、长文本摘要、术语抽取提示词。
- `format.py`：自然语言格式配置解析、文档整理提示词。
- `documents.py`：上传文件图像识别提示词。
- `assistant.py`：编辑助手系统提示词。

服务层只负责传入业务数据，具体模型行为由 prompt 模块集中定义，便于后续调整 AI 输出风格和约束。

## 四、功能模块实现

### 1. 主页与文档入口模块

前端实现文件是 `frontend/src/pages/HomePage.tsx`。主页主要提供文档入口能力：

- 刷新文档列表：调用 `listStoredDocuments()` 获取后端保存的文档摘要。
- 新建文档：调用 `createStoredDocument()` 创建空文档。
- 上传文档：调用 `uploadStoredDocument()`，把文件和视觉模型配置通过 FormData 发给后端。
- 删除文档：调用 `deleteStoredDocument()` 删除文档，并通过自定义事件通知其他页面清理缓存。
- 打开文档：根据用户选择跳转到编辑、翻译或格式整理页面。

后端由 `backend/app/routers/documents.py` 和 `backend/app/services/document_service.py` 支撑。上传文件时，后端先保存原始文件，再把 txt/md/pdf/doc/docx/ppt/pptx 转成图片页，调用视觉模型识别为 Markdown 内容，最后拆分为结构化段落并写入 SQLite。

### 2. 编辑器模块

前端实现文件是 `frontend/src/pages/EditorPage.tsx`。编辑器使用 TipTap 构建富文本/Markdown 编辑体验，依赖 `@tiptap/react` 和多个 TipTap 扩展。项目还定义了本地扩展：

- `frontend/src/extensions/CalloutBlock.ts`：提示块/引用式内容块。
- `frontend/src/extensions/ToggleBlock.ts`：可折叠块。
- `frontend/src/extensions/StructureFold.ts`：结构折叠相关能力。

编辑器模块主要实现：

- 文档内容加载：接收 `AppShell` 传入的 `documentContent`、`documentId` 和段落结构。
- 自动保存：编辑内容变化后延迟调用 `saveStoredDocument()` 或 `saveStoredDocumentParagraphs()`，并向 `AppShell` 回传保存状态。
- 标题同步：标题变化通过 `onTitleChange` 同步到顶部栏和本地缓存。
- Markdown/纯文本复制：将编辑内容导出为 Markdown 或纯文本后写入剪贴板。
- 编辑助手：收集用户消息和上下文，调用 `/api/assistant/chat` 流式获取 AI 回复。
- 助手历史：通过 `getDocumentAssistantHistory()` 和 `saveDocumentAssistantHistory()` 保存当前文档对应的助手对话。

后端助手功能由 `assistant.py` 路由和 `assistant_service.py` 实现。它把前端消息加上 `ASSISTANT_SYSTEM_PROMPT`，再交给 `stream_chat_model()`，最终以模型供应商的 SSE 数据形式返回给前端。

### 3. 翻译模块

前端实现文件是 `frontend/src/pages/TranslatePage.tsx`。翻译页面接收当前文档或用户输入文本，提供：

- 翻译方向：中译英或英译中。
- 显示模式：分栏、段落或句子。
- 翻译风格和额外要求。
- 术语表设置和自动术语抽取。
- 翻译结果复制和保存。
- 文档级翻译状态恢复。

前端调用的主要接口包括：

- `translateTextStream()`：流式翻译。
- `extractTerms()`：术语抽取。
- `getDocumentTranslationState()`：加载某文档的翻译状态。
- `saveDocumentTranslationState()`：保存某文档的翻译结果、chunk、段落对和句子对。

后端实现位于 `translation.py` 路由、`translation_service.py` 和 `translation_state_service.py`。

翻译服务的核心策略是：

1. 先用 `split_translation_units()` 将文本拆成翻译单元。
2. 用 `should_use_long_text_strategy()` 判断是否进入长文本策略。阈值包括文本长度、段落数量和单段长度。
3. 短文本直接构造翻译 prompt 调用模型。
4. 长文本先调用模型生成上下文摘要，再按 chunk 翻译每段内容。
5. 翻译完成后使用 `align_pairs()` 尝试对齐源文和译文段落。
6. 再通过 `sentence_pairs_from_paragraph_pairs()` 在句子数量一致时生成句子级对齐。
7. 流式接口逐步发送 `start`、`context_summary`、`chunk`、`complete` 或 `error` 事件。

翻译状态保存到 SQLite 的 `document_translation_states` 表。这样用户切换页面或重新打开文档后，可以恢复之前的翻译结果。

### 4. 格式整理模块

前端实现文件是 `frontend/src/pages/FormatPage.tsx`。该模块接收文档内容后，提供自然语言格式解析、AI 内容整理和 Word 导出。

前端主要调用：

- `parseFormatPrompt()`：把用户自然语言要求解析成结构化格式配置。
- `organizeFormat()`：让模型把正文整理成标题、二级标题、段落和项目符号块。
- `exportFormatDocx()`：把结构化文档块导出为 Word 文件。

后端由 `format.py` 路由和 `format_service.py` 实现。

格式解析流程是：

1. 前端提交自然语言格式要求和当前配置。
2. 后端通过 `build_format_parse_prompts()` 构造 prompt。
3. 模型返回 JSON。
4. `parse_format_config()` 解析 JSON，并通过 `UI_OPTIONS` 把模型的自由文本归一化成前端下拉选项。
5. 无法归一化的内容放入 `extraRequirements`。

内容整理流程是：

1. 前端提交原文、格式配置和模型配置。
2. 后端调用模型返回 `blocks` 数组。
3. `organize_document()` 校验每个 block 的 `type` 和 `text`。
4. 返回给前端用于预览和导出。

Word 导出流程是：

1. 前端提交标题、blocks 和格式配置。
2. 后端用 `python-docx` 创建 Word 文档。
3. `configure_section()` 设置纸张、页边距、页眉页脚。
4. `add_block()` 根据 block 类型添加标题、段落或项目符号。
5. `apply_run_style()` 和 `apply_paragraph_style()` 设置字体、字号、行距、缩进和对齐。
6. 通过 `StreamingResponse` 返回 docx 文件流。

### 5. 文档管理与段落结构模块

文档管理的后端核心是 `document_service.py`。它不仅保存整篇 Markdown 内容，还维护结构化段落表 `document_paragraphs`。

主要实现包括：

- `markdown_to_paragraph_inputs()`：把 Markdown 拆成标题、heading、paragraph、list、table 等段落输入。
- `paragraphs_to_markdown()`：把结构化段落重新合成为 Markdown。
- `create_document()`：新建文档、计算内容 hash、初始化 RAG 状态。
- `update_document()`：更新标题、正文和术语表，并根据内容 hash 判断 RAG 状态是否变为 `outdated`。
- `update_document_paragraphs()`：按段落保存编辑器结构化内容。
- `delete_document()`：删除文档、段落、助手历史、翻译状态、RAG 索引和上传文件。
- `upload_and_parse_document()`：上传文件识别为 Markdown 后入库。

文档的 `rag_status` 有 `not_indexed`、`indexing`、`indexed`、`outdated`、`failed` 等状态。编辑内容变化后，如果原来已经索引，状态会变成 `outdated`，提醒用户重新索引知识库。

### 6. 知识库与 RAG 问答模块

前端实现文件是 `frontend/src/pages/DocumentsPage.tsx`。它负责：

- 展示文档列表和索引状态。
- 选择多个文档参与问答。
- 调用 `indexStoredDocumentWithRag()` 建立知识库索引。
- 调用 `queryRagStream()` 发起流式问答。
- 展示每轮检索命中的片段和回答内容。
- 保存、重命名、删除知识库问答会话。

后端由 `documents.py` 中的 `/documents/{document_id}/index`、`rag.py` 中的 `/rag/query/stream` 和 `rag_service.py` 实现。

索引流程是：

1. 前端选择文档并传入 RAG 配置。
2. 后端把文档状态标记为 `indexing`。
3. `split_document_paragraph_chunks()` 根据段落结构切 chunk，并把标题上下文附加到段落内容中。
4. `embed_texts()` 根据配置选择本地 sentence-transformers 或 embedding API 生成向量。
5. 向量写入 Chroma，chunk 元数据写入 SQLite 的 `rag_chunks`。
6. 同时写入 FTS5 虚拟表 `rag_chunks_fts`，用于 BM25 全文检索。
7. 索引成功后更新 `last_indexed_at` 和 `rag_status`。

问答流程是：

1. 前端提交问题、文档 id 列表、RAG 配置和聊天模型配置。
2. `retrieve()` 先执行向量召回。
3. 如果配置为 `hybrid`，再执行 FTS5/BM25 检索，并通过 `merge_results()` 合并分数。
4. 如果启用 rerank，`maybe_rerank()` 用词项重合度对召回结果做轻量重排。
5. 取最终 Top 5 片段作为上下文。
6. 后端先通过 SSE 返回 `retrieval` 事件，让前端显示命中文档和片段。
7. 后端把问题和上下文构造成 RAG messages，调用 `stream_chat_model()` 流式生成答案。
8. 前端逐步接收 `chunk` 和 `complete` 事件并更新页面。

这个模块的系统级依赖最多：文档存储提供内容，RAG 设置提供索引参数，Chroma 提供向量检索，SQLite FTS 提供关键词检索，LLM 客户端提供最终回答生成。

### 7. 知识库会话和助手历史模块

会话历史由 `chat_history.py` 路由和 `chat_history_service.py` 实现，数据保存在 SQLite。

知识库问答会话保存到 `knowledge_conversations` 表，字段包括：

- 会话标题。
- 参与问答的文档 id 列表。
- 聊天消息。
- 总检索结果。
- 每一轮单独的检索结果。
- 创建时间和更新时间。

文档编辑助手历史保存到 `document_assistant_messages` 表，按文档 id 关联。这样每篇文档都有自己的助手上下文，不会和其他文档混在一起。

这个模块为编辑器和知识库问答提供“可恢复上下文”，使用户切换页面或重新打开应用后仍能看到历史记录。

### 8. 设置模块

前端实现文件是 `frontend/src/pages/SettingsPage.tsx`。设置页主要管理两类配置：

- 模型配置：服务商、API Key、Base URL、默认模型。
- RAG 配置：embedding 来源、embedding API、本地模型路径、召回策略、rerank 配置。

模型设置会保存到 localStorage，并通过 `writerhub:model-settings-saved` 事件通知 `AppShell` 更新顶部模型状态。设置页还调用 `testFormatModel()` 来测试模型连通性。

后端配置接口 `/api/config/ai` 可以返回后端默认 AI 配置和是否存在 API key，但当前主要运行时配置仍由前端随请求传入。

### 9. Markdown 检测模块

Markdown 检测由 `markdown.py` 路由和 `markdown_service.py` 实现。它不是 AI 功能，而是规则检测。

`markdown_service.py` 定义了一组正则特征，包括：

- 标题。
- 无序列表。
- 有序列表。
- 引用块。
- 围栏代码块。
- 行内代码。
- 加粗、斜体。
- 链接、图片。
- 表格。
- 分隔线。

检测函数返回三个值：是否像 Markdown、命中特征列表和分数。前端可以用它判断导入内容的格式特征。

## 五、模块之间的服务关系

系统级组件和功能模块的关系可以概括为：

- `AppShell` 为所有前端功能提供导航、当前文档、顶部状态、文件选择和缓存。
- `api.ts` 为所有前端功能提供统一后端访问能力。
- `modelSettings.ts` 为翻译、格式整理、编辑助手、文档识别和 RAG 回答提供模型配置。
- `ragSettings.ts` 为知识库索引和检索提供 embedding 与召回配置。
- FastAPI 路由为前端提供稳定接口边界。
- Pydantic schema 为接口提供结构校验和类型约束。
- `llm_client.py` 为所有 AI 模块提供统一模型调用能力。
- `document_service.py` 为编辑、翻译、格式整理和 RAG 提供统一文档数据源。
- SQLite 为文档、段落、历史、翻译状态和 RAG 元数据提供持久化。
- Chroma 为 RAG 提供向量检索。
- FTS5 为 RAG 提供关键词检索。
- prompt 模块为各 AI 功能提供稳定的模型行为约束。

因此，每个功能模块本身并不是孤立实现的，而是复用系统层能力：前端统一调 API，后端统一调服务，服务统一使用模型客户端、存储系统和 prompt 构造。

## 六、典型端到端流程

### 1. 编辑并保存文档

1. 用户在编辑器中修改内容。
2. `EditorPage` 检测到内容变化。
3. 前端延迟触发保存，调用 `saveStoredDocument()` 或 `saveStoredDocumentParagraphs()`。
4. 后端更新 SQLite 中的文档和段落。
5. 后端重新计算内容 hash。
6. 如果文档曾经索引过且内容变化，RAG 状态改为 `outdated`。
7. 前端更新保存状态和本地缓存。

### 2. 上传并识别文档

1. 用户在主页上传文件。
2. 前端把文件和视觉模型配置提交到 `/api/documents/upload`。
3. 后端保存原始文件。
4. 文本文件会被绘制成图片页，PDF 会渲染成图片页，Word/PPT 会先用 LibreOffice 转成 PDF 再渲染成图片页。
5. 后端调用视觉模型逐页识别。
6. 识别结果合并成 Markdown。
7. 后端拆分段落并写入 SQLite。
8. 前端刷新文档列表。

### 3. 翻译长文档

1. 用户在翻译页选择文档和翻译配置。
2. 前端调用 `/api/translate/stream`。
3. 后端判断文本较长，先生成上下文摘要。
4. 后端按 chunk 翻译，每完成一块就发送 SSE 事件。
5. 前端逐块展示译文。
6. 完成后前端可以保存翻译状态。
7. 后端把翻译结果、段落对齐和句子对齐保存到 SQLite。

### 4. 知识库问答

1. 用户先对文档执行索引。
2. 后端把文档拆分为 chunk，生成 embedding，写入 Chroma 和 SQLite FTS。
3. 用户选择文档并提问。
4. 后端做向量召回，必要时合并 BM25 结果。
5. 后端把命中片段先发给前端。
6. 后端把片段作为上下文调用聊天模型。
7. 前端流式显示回答，并保存会话。

## 七、当前实现特点

当前系统的主要特点是：

- 前后端边界清晰：前端负责状态和交互，后端负责业务和数据。
- AI 接入集中：所有模型调用都通过 `llm_client.py`，便于适配不同服务商。
- 文档数据统一：编辑、翻译、格式整理和知识库都围绕同一个文档存储服务工作。
- RAG 同时支持向量检索和关键词检索：Chroma 负责语义召回，SQLite FTS5 负责 BM25。
- 支持流式交互：翻译、助手和知识库问答都能以 SSE 方式逐步返回结果。
- 支持文档级状态恢复：文档内容、翻译状态、助手历史、知识库会话都能持久化。
