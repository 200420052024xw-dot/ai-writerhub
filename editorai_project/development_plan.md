# 开发计划

## 总体策略

项目按“先跑通主流程，再接入真实能力”的方式开发。第一阶段优先完成可运行的前后端骨架和四个核心页面，后续逐步替换 mock 数据为真实 AI、文档解析、检索和导出服务。

技术栈确定为：

- 前端：React + Vite + TypeScript
- 后端：FastAPI + Pydantic + Uvicorn
- 文档处理：`python-docx`、`pypdf`、`openpyxl`
- Markdown：前端使用 Markdown 解析库渲染预览
- AI 能力：后端先封装 mock 服务，后续替换为真实模型调用

## 阶段 0：项目初始化

### 目标

建立可运行的前后端工程结构。

### 前端任务

- 创建 React + Vite + TypeScript 项目。
- 建立基础目录：
  - `src/pages`
  - `src/components`
  - `src/layouts`
  - `src/services`
  - `src/types`
  - `src/styles`
- 配置基础样式、路由和全局布局。

### 后端任务

- 创建 FastAPI 项目。
- 建立基础目录：
  - `app/main.py`
  - `app/routers`
  - `app/schemas`
  - `app/services`
  - `app/storage`
  - `app/core`
- 提供健康检查接口：`GET /api/health`。
- 配置 CORS，允许前端本地开发访问。

### 验收标准

- 前端可以本地启动并显示基础页面。
- 后端可以本地启动并返回健康检查结果。
- 前端可以成功请求后端健康检查接口。

## 阶段 1：统一应用框架和静态 UI

### 目标

基于示例图实现主界面框架和四个页面的静态版本。

### 前端任务

- 实现 `AppShell`：
  - 左侧导航栏
  - 顶部文档栏
  - 主内容区
  - 右侧辅助面板
  - 底部状态栏
- 实现四个页面：
  - 编辑器页面
  - 翻译页面
  - 格式整理页面
  - 文档问答页面
- 实现页面切换和当前导航高亮。
- 使用示例图中的布局、间距、蓝色主色和工具型视觉风格。

### 后端任务

- 暂不实现复杂业务，只保留基础 API。

### 验收标准

- 四个页面都能通过左侧导航进入。
- 页面结构与 `fronted_picture` 示例图一致。
- 页面在常见桌面宽度下无明显布局错位。

## 阶段 2：编辑器核心能力

### 目标

实现文本 / Markdown 编辑的核心体验。

### 前端任务

- 实现 Markdown 编辑区。
- 支持编辑视图、预览视图、分屏视图。
- 接入 Markdown 渲染。
- 自动检测 Markdown 状态。
- 实现字数、字符数、预计阅读时间统计。
- 实现复制：
  - Markdown 原文
  - 普通文本
  - 富文本

### 后端任务

- 提供 Markdown 检测接口：`POST /api/markdown/detect`。
- 第一版可以使用规则判断，例如标题、列表、代码块、链接、表格等语法特征。

### 验收标准

- 用户输入 Markdown 后可实时预览。
- 三种视图切换正常。
- 复制按钮可得到对应格式内容。
- Markdown 检测状态能随内容变化。

## 阶段 3：中英互译

### 目标

实现翻译页面完整前端流程，并通过 FastAPI mock 接口联调。

### 前端任务

- 支持中文转英文、英文转中文。
- 支持翻译方向切换。
- 支持分屏对照、段落交替、句对句对照。
- 实现翻译增强选项 UI。
- 实现术语表 / 记忆库展示。
- 支持复制和导出译文占位操作。

### 后端任务

- 提供翻译接口：`POST /api/translate`。
- 请求包含：
  - 原文
  - 翻译方向
  - 展示模式
  - 增强选项
  - 术语表选项
- 第一版返回 mock 翻译结果，保持响应结构稳定。

### 验收标准

- 前端可以调用 FastAPI 翻译接口。
- 三种展示模式可切换。
- 翻译方向和增强选项能进入请求参数。
- 页面可完整模拟翻译工作流。

## 阶段 4：一句话格式整理和导出

### 目标

实现自然语言格式解析、结构化配置展示、排版预览和导出入口。

### 前端任务

- 实现格式要求输入框。
- 实现智能解析按钮。
- 展示结构化格式配置：
  - 字体
  - 字号
  - 行距
  - 缩进
  - 对齐方式
  - 页边距
  - 页眉页脚
- 实现文档页面预览。
- 实现格式化范围选择。
- 实现导出 Word / PDF 操作入口。
- 展示格式变更记录和格式建议。

### 后端任务

- 提供格式解析接口：`POST /api/format/parse`。
- 提供格式应用接口：`POST /api/format/apply`。
- 提供导出接口：
  - `POST /api/export/docx`
  - `POST /api/export/pdf`
- 第一版格式解析返回 mock 结构化配置。
- Word 导出优先使用 `python-docx`。
- PDF 导出第一版可先返回占位响应，后续再接真实转换。

### 验收标准

- 输入自然语言格式要求后，可以得到结构化配置。
- 点击应用格式后，预览区状态变化。
- Word / PDF 导出接口可被前端调用。
- 导出失败时前端有明确提示。

## 阶段 5：多文档 AI 问答

### 目标

完成多文档上传、解析状态、检索结果、问答和引用展示的完整流程。

### 前端任务

- 实现文档上传区域。
- 展示文档列表和解析状态。
- 支持选择参与问答的文档。
- 实现问题输入和回答展示。
- 回答中展示引用标签。
- 右侧展示检索结果和引用片段。
- 支持复制回答、重新生成回答、查看原文入口。

### 后端任务

- 提供文档上传接口：`POST /api/documents/upload`。
- 提供索引接口：`POST /api/documents/index`。
- 提供文档问答接口：`POST /api/documents/ask`。
- 提供引用片段接口：`GET /api/documents/{document_id}/chunks/{chunk_id}`。
- 第一版解析可先保存文件元信息和 mock chunk。
- 后续接入真实文档解析、分块、向量化和检索。

### 验收标准

- 用户可以上传多个文件并看到状态变化。
- 用户可以选择部分文档参与问答。
- 问答结果包含来源文件、页码或位置、引用片段。
- 检索结果按相关度展示。

## 阶段 6：真实 AI 和文档能力接入

### 目标

将 mock 服务替换为真实 AI、文档解析、检索和导出能力。

### 后端任务

- 接入真实 AI 模型调用。
- 实现文档解析：
  - PDF 文本提取
  - Word 文本提取
  - Markdown 文本读取
  - Excel 表格内容提取
- 实现文档分块和检索索引。
- 实现基于检索结果的问答。
- 实现真实 Word 导出。
- 确定 PDF 导出方案并实现。

### 前端任务

- 替换 mock 状态为真实接口状态。
- 增加加载、错误、重试、空状态。
- 增加上传进度和解析进度展示。

### 验收标准

- 翻译、格式解析、文档问答返回真实 AI 结果。
- 文档问答引用能对应实际文件内容。
- Word / PDF 导出能生成可打开文件。

## 阶段 7：质量完善

### 目标

提升稳定性、可用性和代码质量。

### 任务

- 补充前端组件测试和关键交互测试。
- 补充后端 API 测试。
- 增加统一错误处理。
- 增加请求日志和关键业务日志。
- 优化大文档上传和解析性能。
- 优化移动端或窄屏适配。
- 整理部署文档。

### 验收标准

- 核心接口有基础测试覆盖。
- 前端关键流程可重复验证。
- 常见失败场景有清晰提示。
- 项目具备本地启动和部署说明。

## 优先级

最高优先级：

1. 项目初始化。
2. 四个页面静态 UI。
3. 编辑器 Markdown 编辑和预览。
4. FastAPI mock 接口联调。

第二优先级：

1. 翻译流程。
2. 格式解析流程。
3. 文档问答 mock 流程。

第三优先级：

1. 真实 AI 接入。
2. 真实文档解析和检索。
3. Word / PDF 真实导出。

## 开发原则

- 前端先保证页面和交互完整，再接入真实数据。
- 后端先保证 API 结构稳定，再逐步替换服务实现。
- 所有 AI 能力都通过 FastAPI 后端封装，前端不直接调用模型。
- 文档问答必须保留来源引用，不能只返回纯文本答案。
- 导出功能必须以可预览、可确认的格式配置为前提。
