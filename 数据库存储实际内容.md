## 2. 文档主表

建表结构：

```sql
CREATE TABLE documents (
    id TEXT PRIMARY KEY,
    title TEXT NOT NULL,
    content TEXT NOT NULL,
    content_hash TEXT NOT NULL,
    rag_status TEXT NOT NULL,
    last_saved_at TEXT NOT NULL,
    last_indexed_at TEXT,
    glossary_json TEXT NOT NULL DEFAULT '[]'
);
```

```json
  "title": "Sapper",
  "content_hash": "0dd69611f526a4ed9382c48efbafa7ed8e526cb558439c847126a383aff19e9e",

  "last_indexed_at": null,

```

`documents.content` 不是独立编辑结构，而是把 `document_paragraphs` 按顺序使用空行拼接后的全文快照。功能模块应优先读取 `document_paragraphs`，不应重新按全文猜测段落。

## 3. 结构化段落表

建表结构：

```sql
CREATE TABLE document_paragraphs (
    id TEXT PRIMARY KEY,
    document_id TEXT NOT NULL,
    paragraph_index INTEGER NOT NULL,
    type TEXT NOT NULL,
    level INTEGER NOT NULL,
    content TEXT NOT NULL,
    content_hash TEXT NOT NULL,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| `id` | 段落稳定 ID，翻译、RAG、格式整理都应引用这个 ID |
| `document_id` | 所属文档 ID |
| `paragraph_index` | 段落在文档中的顺序，从 0 开始 |
| `type` | `title`、`heading`、`paragraph`、`list` 或 `table` |
| `level` | 标题或段落所属层级 |
| `content` | 当前段落原文 |
| `content_hash` | 当前段落内容哈希 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

## 7. 当前发现的问题

### 7.1 标题被重复存储

数据库中同时存在：

```text
index 0: type=title, content=Sapper
index 1: type=paragraph, content=Sapper
```

这会造成正文和翻译输入中多出一个普通段落 `Sapper`。

### 7.2 翻译页面自动保存会重建部分段落 ID

在本次只读检查期间，翻译页面触发自动保存后，索引 1、2、3、4、28 的段落 ID 发生了变化，而内容没有变化。

原因是翻译页保存原文时仍可能把编辑后的全文提交给文档更新接口，后端再从 Markdown 全文重新解析段落。重新解析无法稳定继承所有原段落 ID，因此生成了新 ID。

这不符合“段落 ID 稳定、功能模块直接引用存储结构”的目标。

### 7.3 句对句模式与翻译执行模式绑定

只有在点击“开始翻译”之前已经选择“句对句对照”，后端请求的 `display_mode` 才是 `sentence`，模型才会收到 37 个带句子 ID 的结构化单元。

如果先用默认“分屏对照”执行翻译，再切换到“句对句对照”，页面只能尝试把已经完成的段落译文二次切句，不能保证真正的句子 ID 对齐。

### 7.4 翻译结果当前未写入数据库

`document_translation_states` 当前为空。刷新页面或缓存失效后，句子对、段落对和翻译模式都无法从数据库恢复。

## 8. 正确的数据流应当是

```text
document_paragraphs
    -> 段段翻译：直接使用 paragraph_id
    -> 句句翻译：程序按标点拆分，生成 paragraph_id:sentence_index
    -> 模型按相同 ID 返回译文
    -> 按 sentence_index 拼回段落
    -> 页面按 sentence_pairs 或 paragraph_pairs 展示
    -> 翻译状态写入 document_translation_states
```

翻译页修改原文时，应提交完整的结构化段落数组，并保留未改变段落的原 ID；不应只提交拼接后的全文让后端重新推断全部段落。
